## A. 硬冲突 / 错误（🔴 必须修）

### A1. 第 8 节走查的算术有误，且"1.6× 富余"结论建立在错误之上
GDD §8 逐步算式：
```
单发 = 230 ×1.0 ×(1+0.60) ×1.25 ×(1+0.15) = 529
```
- **算术错**：230 × 1.60 × 1.25 × 1.15 = **529** ✅ 这一步对。
- 但下一行：
```
多重射×2 → 1408 / 攻击周期
攻速 1.2/s → DPS ≈ 1408 ×1.2 = 1690
```
- 704 × 2 = 1408，1408 × 1.2 = **1689.6** ✅ 对。
- **但"护甲减伤 ~25% → 实际 DPS ≈ 1268"**：1689.6 × 0.75 = **1267.2** ✅ 对。
- **真正的错**：§3.1 公式里 `crit_factor = 1 + 暴击率×(暴伤−1)`，暴伤=210% 即 2.1，所以因子 = 1 + 0.30×(2.1−1) = **1.33** ✅。**但 GDD §3.2 没定义"基础暴伤"是多少**——走查用了 210%（=1.5 基础 + 词条），可 §3.1 / CombatStats 默认 crit_dmg=1.5（150%）。**基础暴伤在文档里未声明**，读者无法复现 210% 从哪来。需补"英雄基础暴伤 = 150%"。

> 这是 B-1 walkthrough 测试要锁定的回归基线，必须先在 GDD 把基础暴伤写死，否则测试无依据。

### A1b. 【修正后新发现，🔴 最重要】§8 走查的 ATK=230 / DPS=1268 整个不可复现
修 A1（补基础暴伤）后顺查发现：原走查声称"5 个羁绊撑到 ATK 230"，但羁绊给的 ATK 加成只有荒古圣体+15% + 遮天2件套+20% = +35%，即 50×1.35 = **67.5**，不是 230（230 需 +360% ATK）。连带 DPS 1268 / "1.6× 富余"全部无来源。
**已修正**为可追溯的 67.5 ATK → 实际 DPS ≈ 373，仅占第 15 波所需 787 的 47%。
**这暴露真问题**：纯羁绊+单技能词条的中期 build 打不过第 15 波——说明基础数值(羁绊层)偏弱 或 敌人曲线(1.18)偏陡。**留作 B-1 首要校准项**（降曲线 or 升羁绊）。这正是数值验证脚本存在的价值：用代码把这种"拍脑袋数字"揪出来。

### A2. §10 决策记录与"剩余开放问题"编号断裂
- "已确认决策"列了 1–13 条。
- "剩余开放问题"却从 **10** 开始编号 → 与决策 10（Boss debuff）撞号，且跳过了语义。
- 应改为独立编号（如 Q1–Q5）或接续 14 开始。

### A3. 技术文档与 GDD 对"boss_total_hp_share=0.4"的理解冲突
- GDD §3.3：`Boss 波血量 ≈ 该波总血量的 40%，单独结算`。
- `waves.yaml`：`boss.total_hp_share: 0.4   # Boss 本体血量占该波总血量比例`。
- **歧义**：这 40% 是"Boss 单体 = 该波总血量的 40%"（那小怪占另外 60%？总血量怎么算），还是"该波总血量里有 40% 归 Boss"？
- §3.3 表里 wave 10 总血量 10,212、所需 DPS 292——如果 Boss 占 40% 即 ~4,085 血单独结算，那小怪总血量 = 6,127，但所需 DPS 是按"总血量/时长"算的，没扣掉 Boss 单独结算的影响。**Boss 单独结算 vs 纳入总血量，二选一，当前同时存在**。

> 建议：Boss 波的"总血量"明确 = 小怪总血量 + Boss 血量，所需 DPS 按此总血量算；"单独结算"只指"Boss 有独立血条显示"，不是单独算 DPS 需求。改 GDD 措辞。

### A4. `data/equipment.yaml` 里程碑与 GDD §5.3 不一致
- GDD §5.3 表：+1/+2/+3/+6/+9。
- `equipment.yaml`：`milestones: [3, 6, 9]` ✅ 一致。
- 但 §5.3 表把 +3/+6/+9 都列了"抽词条"，而 `per_level_income` 只定义了 gold_per_sec / per_kill 两种固定收益，**没说奇数级/偶数级如何分配**（注释说"奇数级 +1 金/秒，偶数级 +0.5 杀敌金"，但 +2 表里写的是"+0.5 杀敌金"✅，+1 写"+1 金/秒"✅，+3 写"+1 金/秒"——但 +3 是奇数应 +1 金/秒 ✅）。**实际一致，但数据没显式表达"奇偶分配规则"**，loader 无法直接用。需把每级收益写成显式数组。

## B. 设计漏洞 / 张力不足（🟡 应修）

### B1. 🔴→🟡 "换 debuff 40 金"与"贪婪之噬 debuff"形成负面耦合
- §13：换 debuff 花 40 金。
- §12.2：贪婪之噬 = 本波金币获取 −50%。
- **漏洞**：如果 Boss 波前就决定要"换/删"，钱是在**上一波攒的**，贪婪之噬只影响"本波（Boss 波）金币获取"，**不影响构筑大厅花销**。✅ 这个其实没冲突。
- **但反过来**：贪婪之噬 Boss 波打完给的金币 −50%，会**削弱下一章攒钱买道具的能力**。这是设计意图还是 bug？建议明确"贪婪之噬只影响战斗内金币掉落，不影响大厅已持有金币"，并在文档写清。

### B2. "禁疗之印"对回血流的克制可能过强（一击致命）
- §12.4 回血流 = 吸血 3% + 复苏 2%/5s + 圣泉 1%/击杀 + 法天象地 30% 上限。
- §12.2 禁疗之印 = 本波玩家回血**全部失效**。
- **张力问题**：如果玩家 build 重度依赖回血生存（没堆减伤/血量），禁疗之印 = 直接暴毙。这违反 §12.1 原则 3"痛但不致命"。
- **建议**：禁疗之印改为"回血效果 −70%"（强削弱但留余地），或保证回血流同时总有"非回血"生存手段。当前回血流与减伤/血量流是否互斥需澄清。

### B3. "断舍离"诅咒 + 羁绊抽取逻辑冲突
- 断舍离：羁绊池上限 −2（→ 8）。
- §6.1：池子上限 10。
- **漏洞**：如果玩家先抽满 10 个羁绊，再拿"断舍离"，池子变 8 → 多出的 2 个怎么办？强制丢弃？哪个？玩家选？
- 需定义：拿断舍离时若超限，让玩家选丢弃哪些；或断舍离只在"未满池"时可拿。

### B4. 联动触发条件"bond_owned vs bond_devoured_set"混用，语义不清
- §7 联动表里：圣体真伤用 `bond_owned`（拥有即可），天帝之拳用 `bond_devoured_set`（需吞噬）。
- 但 `data/synergies.yaml` 里 `saint_true_dmg` 用 `bond_owned: ancient_saint_body`，`zhutian_emperor_fist` 用 `bond_devoured_set: zhutian`。
- **漏洞**：吞噬后羁绊从"羁绊池"进"已吞噬池"——那"拥有(bond_owned)"指当前池里，还是包括已吞噬的？需明确定义这两个状态的判定范围。否则圣体真伤在"圣体被吞噬后"还触不触发？

### B5. 大招能量"30–45s 蓄满"与单波时长不匹配
- §11：大招 30–45s 蓄满。
- §3.3：wave 1 时长 25s，wave 30 时长 55s。
- **问题**：前 5 波（25–30s）大招可能**蓄不满一次**，玩家无大招可用 → 与"手动大招给掌控感"的设计意图冲突。
- 建议：蓄满时间随波次动态调整，或前几波降低门槛（如 20s）。验收时检查"每波至少能放 1 次大招"。

### B6. "每波 1.5–2 次 Rogue 操作"经济目标 vs 实际数值对不上
粗算 wave 5（时长 30s）收入：
- 被动 5×30 = 150；击杀 ~16 怪×2 = 32；清场 60；小精英(波5) 40。**合计 ≈ 282 金**。
- 一次抽羁绊 30 金 + 一次升装备 ~19 金 = 49 金 → 282/49 ≈ **5.7 次操作**。
- **远超"1.5–2 次"目标**。意味着要么收入太高、要么目标定低了。
- 这是 B-2 要校准的，但 GDD §3.4 的目标值与初值明显量级不符，应先标注"待 B-2 校准，当前收入偏高"。

## C. 优化 / 澄清（🔵 建议）

### C1. 暴击率封顶 100% 但没有"暴击抗性"对冲
后期若玩家堆到 100% 暴击，致盲之眼(暴伤−50%)是唯一软克制。建议加一个 debuff 或敌人词缀"暴击率 −X%"，让暴击流也有被针对的空间。非必须。

### C2. §3.1 公式缺少"属性伤害"独立乘区的正式位置
§3.1 主公式没有 elemental 项，但 §3.2 说"属性伤害=独立乘区"。应把 `elemental_mult` 补进主公式，否则实现时会漏。

### C3. 心流文档 §1 自检表已过时
心流 §1 表写"3 选 1 是固定的，缺刷新"——但这已经在 GDD §4.4 决策补上了。心流文档应更新自检状态（2/4/5/6 现在部分已补），否则读者以为还没做。

### C4. "局外 meta 成长"悬而未决影响面较大
开放问题里挂着，但 meta 成长**是否做**会影响：失败是否保留资源、是否有局外升级、服务端 schema。建议尽早决策，哪怕先定"首版不做 meta，失败清零"。

### C5. IP 换皮决策未定，但数据已用参照 IP 名
`bonds.yaml` 用了"遮天/风云/黑神话"。若最终换皮，数据要全改。建议数据层用中性 id（已做：`zhutian/fengyun/blackmyth`），但**展示名**集中到一个 i18n 层，方便换皮时只改文案。

## D. 总评

设计整体**结构清晰、职责分层（技能=机制/装备=经济/羁绊=数值）非常漂亮**，联动引擎数据驱动的设计前瞻性强。Boss debuff + 提前预告 + 换/删两档这套闭环很扎实。

**主要风险集中在数值层**（A1 基础暴伤未定义、A3 Boss 血量歧义、B6 经济目标与初值差 3 倍）——这些恰好是 B-1/B-2 数值脚本要解决的，所以现在的状态是"设计有洞，但工具已在路上"。建议**先修 A 类（硬冲突）再写代码**，否则脚本会锁定错误的基线。

优先级建议：A1 → A3 → A4 → B6 → B2 → B5 → 其余。

## E. B-2 经济模拟新发现（2026-07-03，1000 局蒙特卡洛）

### E1. 🟡 B6 真因不是"收入偏高"，是"缺金币回收出口"（saturation_crash）

原 GDD §3.4 B6 标注假设"收入偏高 → 5.7 次/波"。B-2 模拟推翻此假设：

| 指标 | 模拟结果 | 目标 | 结论 |
|---|---|---|---|
| 早期(1–4波)操作数 | 7 次/波 | 1.5–2 | 偏高（可微调收入） |
| 后期(16–30波)操作数 | **1 次/波** | 1.5–2 | 崩盘 |
| 终局囤积金币 | **9634** | 不应囤积 | 严重 |
| 重投率 | **3%** | 30–50% | 几乎不刷新 |
| 稀有度分布 | 53/32/10/5 | 60/30/8/2 | ✅ 接近 |

**真因**：羁绊池(10)+装备(+9)饱和后，玩家**无事可买**，金币无限囤积。早期看似"操作多"是因为消费空间充足，一旦填满就断崖。

**建议（待决策）**：加金币出口——① 装备满级精炼词条 ② 羁绊重投更激进 ③ 吞噬消耗金币 ④ 重投上限随波次提升。同时 base_passive 5→3 微调早期。

**已落实**：GDD §3.4 B6 注已更新为校准结果；站点 /economy.html 可视化呈现。
